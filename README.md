## рҹҢҗ What is FedCare AI?

**FedCare AI** is a full-stack, production-grade **Federated Learning (FL)** platform purpose-built for the healthcare ecosystem. In traditional machine learning, you must centralize all data вҖ” an impossible ask for hospitals bound by HIPAA, GDPR, and patient confidentiality laws.

FedCare AI flips this model on its head:

```
Traditional ML:       Hospital A в”Җв”Җв”җ
                      Hospital B в”Җв”Җв”јв”Җв”Җв–ә Central Server (has ALL patient data) в–ә Model
                      Hospital C в”Җв”Җв”ҳ

FedCare AI:           Hospital A в”Җв”Җв–ә Trains locally в”Җв”Җв”җ
                      Hospital B в”Җв”Җв–ә Trains locally в”Җв”Җв”јв”Җв”Җв–ә Aggregates only WEIGHTS в–ә Global Model
                      Hospital C в”Җв”Җв–ә Trains locally в”Җв”Җв”ҳ
                                           вҶ‘
                                  No patient data ever leaves
```

Each hospital trains **XGBoost models** on its own premises. Only **model weight updates** are sent to the central server for **FedAvg aggregation** вҖ” patient records stay private, forever.

### рҹ”‘ Core Capabilities

<details>
<summary><b>рҹҸҘ Federated Training Pipeline</b></summary>
<br/>

- **Multi-hospital coordination** вҖ” Hospitals join disease-specific servers and train on their own local CSV/TXT datasets
- **FedAvg aggregation** вҖ” Server aggregates local model weights into a single global model each round
- **Resilient training** вҖ” Handles class-imbalanced datasets, single-class splits, and tiny datasets gracefully with automatic fallback injection
- **Real-time progress** вҖ” Live training logs streamed to the dashboard with polling-based status updates
- **Full metric suite** вҖ” Accuracy, F1, Precision, Recall, AUC-ROC, and log-loss captured every round

</details>

<details>
<summary><b>рҹ“Ҳ Live Training Curve Visualization</b></summary>
<br/>

- **Dual-axis Chart.js plots** вҖ” Accuracy (blue, left axis) vs Loss (red, right axis) rendered per boosting round
- **100-round granularity** вҖ” Full XGBoost internal history (`logloss` + `error`) captured via `evals_result()` and embedded in the training log
- **Auto-extraction** вҖ” Frontend parses embedded `__HISTORY_JSON__` from DB logs to populate charts post-training
- **Live shimmer bar** вҖ” Indeterminate animated progress bar during active training

</details>

<details>
<summary><b>рҹ§  Explainable AI (XAI)</b></summary>
<br/>

- **SHAP integration** вҖ” Generates feature importance scores for every prediction
- **Waterfall plots** вҖ” Visual per-patient SHAP breakdown showing which features drove the diagnosis
- **Global feature ranking** вҖ” Bar chart of mean SHAP values across the dataset
- **Grad-CAM** вҖ” Gradient-weighted class activation mapping (for imaging-based models)

</details>

<details>
<summary><b>рҹ”® Real-Time Prediction Engine</b></summary>
<br/>

- **Dynamic input forms** вҖ” Auto-generated from the server's feature schema
- **Probability output** вҖ” Confidence score and positive/negative probability shown alongside prediction
- **XGBoost inference** вҖ” Loads the latest saved global model and runs `predict_proba` in real-time
- **Feature validation** вҖ” Validates inputs against the exact trained feature columns

</details>

<details>
<summary><b>рҹҺЁ Premium Glassmorphism Dashboard</b></summary>
<br/>

- **Dark mode by default** вҖ” Deep navy/indigo color palette with luminous accents
- **Glassmorphism panels** вҖ” Frosted-glass cards with backdrop-filter blur effects
- **Animated topology** вҖ” Visual federated network graph showing server вҶ” hospital connections
- **Micro-animations** вҖ” Pulse effects, hover transitions, shimmer loaders throughout

## рҹ—әпёҸ Platform Walkthrough

### Step 1: Login

| Role | What They Can Do |
|:---|:---|
| **Admin** | Create disease servers, approve hospitals, trigger training, view all metrics |
| **Hospital** | Upload datasets, join servers, view own training results, run predictions |

> рҹ’Ў For demo purposes, any hospital name + any password works via the universal demo login.

---

### Step 2: Create a Disease Server (Admin)

Go to **Servers вҶ’ Create New Server** and configure:
- Disease type (e.g. Diabetes, Heart Disease)
- Target column name (e.g. `Outcome`)
- Number of federated rounds

---

### Step 3: Hospitals Join & Upload Data

Hospitals navigate to the server, click **Join**, and upload their local CSV/TXT dataset. The system auto-detects feature columns and the target variable.

---

### Step 4: Train the Model

The admin clicks **вҡЎ Train XGBoost Model**. The platform:

1. Loads all hospital datasets
2. Runs local XGBoost training (200 estimators, stratified splits)
3. Aggregates weights via **FedAvg**
4. Saves the global model to `saved_models/`
5. Logs accuracy, F1, AUC-ROC, and full loss/accuracy curves per boosting round

---

### Step 5: Visualize Results

After training, the **Training Curve** section renders:

- рҹ“ҳ **Blue line** вҖ” Validation accuracy (%) rising per boosting round
- рҹ“• **Red dashed line** вҖ” Log-loss decreasing per boosting round
- рҹ“Ӣ **Summary table** вҖ” Round-by-round metric breakdown

---

### Step 6: Run Predictions & Explain

Navigate to **Predictions** вҶ’ enter patient data вҶ’ get real-time diagnosis with confidence score.

Go to **Explainability** вҶ’ view SHAP waterfall charts showing exactly *which* features drove the model's decision for each patient.

**Key Privacy Guarantee:** Raw patient records **never** leave the hospital. Only floating-point weight arrays are transmitted.
